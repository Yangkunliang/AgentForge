## 0. 背景与问题陈述

### 0.1 现状

AgentForge 目前已具备：

- ✅ 多 Agent 协同框架（Contract Net + Harness 六层）
- ✅ Skill 引擎（ReAct 循环，天气 Skill 内置）
- ✅ 流式聊天 UI（SSE + Vue 3）
- ✅ CLAW 集成能力层（L1 http_request Skill，MCP 客户端基础设施）

**但缺少一个关键层**：面向平台终端用户的「产品体验层」——用户打开 AgentForge 后不知道能做什么、怎么用、从哪里开始。

### 0.2 核心问题

> **能力再强，如果用户不知道怎么触发，就等于没有。**

现有快捷功能（SQL 优化、代码审查、安全检查…）只是「快速输入一句话」的浅层封装，存在三个根本问题：

1. **功能孤立**：各个快捷功能之间没有关联，用户完成一步后不知道下一步是什么
2. **流程缺失**：全栈开发的工作是一条完整流水线（需求→架构→开发→测试），现有设计没有体现这一点
3. **上下文丢失**：每次对话都是全新的，Agent 不了解用户的项目背景，生成的内容和实际代码库脱节

### 0.3 目标

设计一套面向全栈开发工程师的 **Agent 交互体验**，覆盖：

1. **项目管理**：用户在平台内管理自己的代码库，作为 Agent 对话的上下文基础
2. **意图路由**：用户通过「需求类型」模式选择器告知 Agent 当前意图，Agent 动态规划流水线阶段
3. **阶段感知对话**：对话界面展示当前所在阶段、阶段产物和下一步建议
4. **快捷动作体系**：每个阶段有对应的快捷动作，每个动作都有明确的产物输出

## 1. 目标用户

### 1.1 用户画像

AgentForge 的终端用户是**全栈开发工程师**，不是开发 AgentForge 本身的工程师。

| 类型 | 描述 | 典型场景 |
|------|------|---------|
| **独立开发者** | 一人维护完整产品，前后端、运维都要做 | SaaS 产品迭代、个人项目开发 |
| **创业团队成员** | 小团队（3-10人），职责边界模糊 | 快速验证功能、全链路参与 |
| **外包接单开发者** | 同时维护多个客户项目 | 快速切换项目上下文、多项目并行 |

### 1.2 用户核心痛点

| 痛点 | 具体表现 |
|------|---------|
| **流程碎片化** | 收到需求后不知道从哪里开始，每个步骤都要手动想 |
| **上下文切换成本高** | 和 AI 对话时需要反复解释项目背景 |
| **产物不可复用** | AI 生成的内容散落在对话里，无法和项目文件关联 |
| **需求类型混淆** | 不管是新功能还是改 Bug，都用同样的方式和 AI 交互，效率低 |

### 1.3 用户故事

```
US-01  作为全栈开发工程师，我希望把我的项目目录挂载到 AgentForge，
       这样 Agent 在给我建议时能参考我真实的代码结构，而不是凭空生成。

US-02  作为全栈开发工程师，我希望告诉 Agent「这是一个新功能需求」或
       「这只是个 UI 调整」，这样它不需要每次都让我走完整的流程。

US-03  作为全栈开发工程师，我希望每次对话产出的文档、代码都和我的项目
       关联起来，下次打开还能继续，不是一次性的聊天记录。

US-04  作为外包开发者，我希望在 AgentForge 里管理多个客户项目，
       快速切换上下文，不需要每次都重新介绍项目背景。

US-05  作为全栈开发工程师，我希望对话界面告诉我现在在哪个阶段、
       产出了什么、下一步该做什么，而不是一大段文字让我自己理解。
```

## 2. 产品模型

### 2.1 核心概念

| 概念 | 定义 | 类比 |
|------|------|------|
| **项目（Project）** | 用户的一个代码库/产品，是平台的一等公民 | GitHub Repository |
| **目录挂载（Mount）** | 用户授权 Agent 访问本地目录或远程仓库 | VS Code 打开文件夹 |
| **会话（Session）** | 归属于某个项目的一次对话，产物与项目绑定 | GitHub Issue |
| **需求类型（Intent）** | 用户当前需求的分类，决定流水线走哪几步 | Git commit type |
| **阶段（Stage）** | 流水线中的一个步骤，有明确的输入、产物、格式 | CI/CD Pipeline Step |
| **产物（Artifact）** | 阶段的输出文件，如 PRD.md、Schema.sql、组件代码 | Build Artifact |

### 2.2 数据关系

```
User
 └── Project（1:N）
      ├── Mount（1:N）           # 挂载的目录/仓库
      ├── Session（1:N）         # 对话会话
      │    ├── Message（1:N）    # 消息记录
      │    └── Artifact（1:N）   # 产物文件
      └── ProjectMeta            # 项目元信息（技术栈标签、角色等）
```

### 2.3 项目生命周期

```
注册 → 创建项目 → 挂载目录 → 开始对话 → 选择需求类型
  → Agent 规划阶段 → 逐阶段执行 → 产出 Artifact → 归档到项目
```

## 3. 功能规格

### 3.1 项目管理页（My Projects）

#### 3.1.1 页面定位

用户注册后的入口页，也是日常使用的起点。**新用户的第一步是创建项目，不是直接开始对话。**

#### 3.1.2 空状态

```
┌─────────────────────────────────────────┐
│                                         │
│        🗂  还没有项目                    │
│                                         │
│   添加你的第一个项目，让 Agent 了解      │
│   你的代码库，告别重复解释背景           │
│                                         │
│        [+ 创建第一个项目]               │
│                                         │
└─────────────────────────────────────────┘
```

#### 3.1.3 项目列表

每个项目卡片展示：

| 字段 | 说明 |
|------|------|
| 项目名称 | 用户自定义（如「我的电商后端」）|
| 技术栈标签 | Vue 3 / FastAPI / PostgreSQL 等 |
| 挂载状态 | 已连接 / 未连接 / 连接中 |
| 最近活跃 | 最后一次会话时间 |
| 快速入口 | 「继续上次对话」/ 「新建对话」|

#### 3.1.4 创建项目流程

**Step 1 — 基本信息**

```
项目名称：[___________________________]  ← 用户自定义，如"电商后端"
项目描述：[___________________________]  ← 可选，一句话说明
```

**Step 2 — 挂载代码库**

提供四种方式，按推荐顺序展示：

```
选择代码库访问方式：

● 本地目录（推荐）
  通过 AgentForge CLI 或桌面客户端授权访问
  ┌─ CLI 方式 ──────────────────────────────┐
  │  $ agentforge mount ~/work/my-project  │
  │  ✅ 已检测到本地 Agent Bridge 运行中    │
  └────────────────────────────────────────┘
  [选择目录]

○ GitHub 仓库
  通过 OAuth 授权，无需本地环境
  [连接 GitHub]

○ 手动上传文件
  上传关键文件作为上下文（兜底方案）
  [上传文件]
```

**Step 3 — 技术栈标签**

```
技术栈（选择或输入）：
[Vue 3 ×] [FastAPI ×] [PostgreSQL ×] [+ 添加]

项目角色（当有多目录时使用）：
● 主项目   ○ 参考项目   ○ 文档库
```

**Step 4 — 完成**

```
✅ 项目创建成功

  「我的电商后端」已就绪
  已挂载：~/work/shop-api（主项目）
  技术栈：FastAPI · PostgreSQL · Redis

  [开始第一次对话 →]
```

#### 3.1.5 多目录管理

- 每个项目支持挂载多个目录，**上限 3 个**
- 每个目录需标注角色：主项目 / 参考项目 / 文档库
- 同名文件来自不同目录时，UI 显示完整相对路径加以区分
- 目录连接状态实时显示（Agent Bridge 是否在运行）

#### 3.2.2 需求类型选择器

位于输入框上方，**4 个模式**，用户打开页面第一眼就看到，明确当前的工作上下文：

| 模式 | 图标 | 触发信号 | 流水线长度 |
|------|------|---------|-----------|
| 全新功能 | ✨ | 新实体/新表/新路由/跨模块 | 8 步（完整）|
| 迭代优化 | 🔄 | 改现有逻辑/局部改动 | 5 步 |
| UI 调整 | 🎨 | 只改前端/不动接口 | 3 步 |
| Bug / 重构 | 🐛 | 报错/性能问题/代码坏味道 | 4 步 |

**切换联动**：选择模式后，以下三个区域同步更新：
1. 阶段预览条（显示本次要走哪几步）
2. 输入框 placeholder（根据模式给出对应的输入提示）
3. 快捷动作栏（每个模式高亮最关键的起始动作）

#### 3.2.3 阶段预览条

位于需求类型选择器下方，实时展示当前模式下的执行路径：

```
# 全新功能模式
需求分析 › 架构设计 › DB & API › 任务拆解 › UI 原型 › 后端开发 › 前端开发 › 测试交付

# 迭代优化模式
需求 Diff › 影响评估* › 后端开发 › 前端开发* › 回归测试
                                              跳过：完整PRD · 架构 · UI原型 · 任务拆解

# UI 调整模式
原型 Diff › 前端开发 › 视觉验收
                                跳过：需求分析 · 架构 · DB · API · 后端 · 任务拆解

# Bug / 重构模式
问题定位 › 影响范围分析 › 修复 › 回归测试
```

> `*` 为可选阶段，Agent 根据需求自动判断，用户也可手动跳过。

#### 3.2.4 上下文 Chips

位于输入框顶部工具栏，展示当前挂载的上下文：

```
上下文：[✓ main 分支] [✓ PRD-CLAW.md] [engine.py] [+ 添加上下文]
```

- **默认激活**：项目的主目录 + 最近修改的 PRD 文档
- **可点击切换**：点击 chip 切换激活/停用状态
- **+ 添加**：从项目文件树中选择具体文件，或切换到其他挂载目录
- **显示规则**：同名文件显示完整相对路径（如 `shop-api/main.py`）

### 3.4 Agent Bridge（本地访问工具）

#### 3.4.1 CLI 工具

```bash
# 安装
npm install -g agentforge-cli
# 或
pip install agentforge-cli

# 挂载目录
agentforge mount ~/work/my-project

# 查看已挂载目录
agentforge mounts

# 卸载目录
agentforge unmount ~/work/my-project

# 查看 Bridge 运行状态
agentforge status
```

#### 3.4.2 Bridge 工作原理

```
AgentForge Web                      用户本地
─────────────────                   ──────────────────
[对话界面]                           [agentforge CLI]
    │                                     │
    │   WebSocket                         │
    └──────────────────────────────────── ┘
                                          │
                                    [filesystem MCP]
                                          │
                                   ~/work/my-project/
```

- CLI 在本地启动一个轻量进程，暴露 filesystem MCP Server
- Web 端通过 WebSocket 与之通信，读写本地文件
- **用户主动授权**：只有 `agentforge mount` 明确指定的目录才可访问
- **断开即失效**：CLI 进程退出后，Web 端自动感知连接断开

#### 3.4.3 GitHub OAuth 方案（无需本地环境）

对于只需要「读代码 + 生成代码」的场景：

1. 用户在项目设置中点击「连接 GitHub」
2. OAuth 授权后，AgentForge 获取仓库读取权限
3. Agent 可以读取指定分支的文件内容
4. 生成的代码通过 AgentForge 界面复制，或通过 GitHub App 创建 PR

## 4. 流水线执行规格

### 4.1 需求类型与阶段映射

#### 全新功能（8 步）

| # | 阶段 | 接收 | 产物 | 格式 | 人工介入 |
|---|------|------|------|------|---------|
| 1 | 需求分析 | 一句话需求、参考文档 | PRD.md、用户故事 | Markdown | **必须确认** |
| 2 | 架构设计 | 已确认 PRD | 架构图、技术选型 | Mermaid + Markdown | **必须确认** |
| 3 | DB & API 设计 | 架构文档 | ER 图、DDL、API-SPEC.md | SQL + OpenAPI YAML | 可选确认 |
| 4 | 任务拆解 | PRD + 架构 | Sprint 计划、任务列表 | Markdown / CSV | 可选确认 |
| 5 | UI 原型 | 用户故事、设计参考 | 可交互 HTML 原型 | HTML | **必须确认** |
| 6 | 后端开发 | API-SPEC + Schema | 路由、Service、Model | Python 源文件 | Review 建议 |
| 7 | 前端开发 | UI 原型、API 文档 | Vue 3 组件、页面 | .vue + .scss | Review 建议 |
| 8 | 测试交付 | 源代码、验收标准 | 测试文件、README | pytest + Markdown | 覆盖率确认 |

#### 迭代优化（5 步）

| # | 阶段 | 产物 | 必须？|
|---|------|------|------|
| 1 | 需求 Diff | 与已有功能的差异描述 | ✅ |
| 2 | 影响评估 | 改动文件列表、风险点 | ✅（必须确认）|
| 3 | 后端开发 | 改动后的 Python 文件 | ✅ |
| 4 | 前端开发 | 改动后的 Vue 组件 | 可选 |
| 5 | 回归测试 | 针对改动点的测试文件 | ✅ |

#### UI 调整（3 步）

| # | 阶段 | 产物 | 必须？|
|---|------|------|------|
| 1 | 原型 Diff | 与现有 UI 的差异说明 | ✅ |
| 2 | 前端开发 | 改动后的 Vue 组件 + 样式 | ✅ |
| 3 | 视觉验收 | 渲染预览截图 | ✅（用户目测确认）|

#### Bug / 重构（4 步）

| # | 阶段 | 产物 | 必须？|
|---|------|------|------|
| 1 | 问题定位 | 根因分析报告 | ✅ |
| 2 | 影响范围分析 | 受影响文件 + 风险评估 | ✅（必须确认）|
| 3 | 修复 / 重构 | 改动后的源文件 | ✅ |
| 4 | 回归测试 | 测试文件 + 覆盖率报告 | ✅ |

### 4.2 人工介入点汇总

全平台共 **3 个必须介入点**，其余由 Agent 自动推进：

| 介入点 | 时机 | 原因 |
|--------|------|------|
| **PRD 确认** | 需求分析完成后 | 所有后续阶段的基础，错了会放大 |
| **技术选型确认** | 架构设计完成后 | 是否引入新中间件是主观决策，影响工作量倍数 |
| **改动范围确认** | 迭代优化 / Bug 修复的影响评估后 | 防止改出副作用，修改前先看清楚 |

### 4.3 Agent 意图分类逻辑

收到用户需求后（如果用户没有手动选择模式），Agent 通过以下信号自动分类：

| 信号 | 结果 |
|------|------|
| 涉及新实体 / 新数据库表 | → 全新功能 |
| 涉及新的 API 路由 | → 全新功能 |
| 改动跨越 3 个以上模块 | → 全新功能 |
| 只改现有逻辑，不加新表 | → 迭代优化 |
| 只改前端文件，明确不动接口 | → UI 调整 |
| 包含报错信息 / Stack Trace | → Bug 修复 |
| 包含「重构」「性能优化」「代码质量」 | → Bug / 重构 |

## 5. 技术实现指引

### 5.1 新增数据模型

在现有 9 张表基础上，新增以下实体（需更新 `docs/tech-design/DATABASE.md`）：

```sql
-- 项目
CREATE TABLE projects (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id     UUID NOT NULL REFERENCES users(id),
    name        VARCHAR(100) NOT NULL,
    description TEXT,
    created_at  TIMESTAMPTZ DEFAULT now(),
    updated_at  TIMESTAMPTZ DEFAULT now()
);

-- 目录挂载
CREATE TABLE project_mounts (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    project_id   UUID NOT NULL REFERENCES projects(id),
    mount_type   VARCHAR(20) NOT NULL,  -- 'local' | 'github' | 'upload'
    path         VARCHAR(500),          -- 本地路径 or GitHub repo
    display_name VARCHAR(100),          -- 用户友好名称
    role         VARCHAR(20) DEFAULT 'primary', -- 'primary' | 'reference' | 'docs'
    status       VARCHAR(20) DEFAULT 'disconnected',
    tech_tags    JSONB DEFAULT '[]',
    created_at   TIMESTAMPTZ DEFAULT now()
);

-- 产物
CREATE TABLE artifacts (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    session_id  UUID NOT NULL REFERENCES sessions(id),
    project_id  UUID NOT NULL REFERENCES projects(id),
    stage       VARCHAR(50) NOT NULL,   -- 'prd' | 'architecture' | 'code' | 'test' ...
    name        VARCHAR(200) NOT NULL,
    content     TEXT,
    file_type   VARCHAR(50),            -- 'markdown' | 'python' | 'vue' | 'sql' ...
    created_at  TIMESTAMPTZ DEFAULT now()
);
```

Sessions 表新增字段：

```sql
ALTER TABLE sessions ADD COLUMN project_id UUID REFERENCES projects(id);
ALTER TABLE sessions ADD COLUMN intent_type VARCHAR(20);  -- 'new_feature' | 'iteration' | 'ui' | 'bugfix'
ALTER TABLE sessions ADD COLUMN current_stage VARCHAR(50);
ALTER TABLE sessions ADD COLUMN stage_history JSONB DEFAULT '[]';
```

## 9. 开放问题

| 问题 | 背景 | 决策建议 |
|------|------|---------|
| Bridge 技术选型：Electron vs Tauri vs 纯 CLI？ | Electron 成熟但重，Tauri 轻但复杂，纯 CLI 最轻 | **MVP 先做纯 CLI**，后续看用户反馈再决定是否做 GUI |
| 意图分类是否调用 LLM？ | 规则分类快但不准，LLM 分类准但慢 | **规则优先**，置信度低时 LLM 兜底 |
| 产物存储：存数据库还是文件系统？ | 数据库查询方便，文件系统可直接 diff | **MVP 存数据库 content 字段**，后续考虑对象存储 |
| 多人协作是否进入 MVP 范围？ | 项目级别的多人协作 | **MVP 不做**，单用户场景先跑通 |
