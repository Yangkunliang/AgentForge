# 核心能力增强产品需求

**日期**：2026-07-08
**目标用户**：使用 AgentForge 开发自己项目的全栈开发工程师

## 背景

TASK-012 至 TASK-019 已完成核心开发闭环 MVP。用户现在可以创建项目、授权本地代码库、生成阶段产物、人工确认，并把 Artifact 写回本地目录。

下一阶段不是只选择交互或服务端中的一个方向，而是按优先级连续推进，让平台从“能跑通”变成“可信、清楚、可扩展”。

## 用户故事

### CDW-08：可信写回

作为全栈开发工程师，我希望 AgentForge 在写回我的代码库前确认目标文件仍是我预览时看到的状态，这样不会覆盖我或队友刚刚做的修改。

### CDW-09：下一步动作清楚

作为全栈开发工程师，我希望在 Project 和 Chat 里清楚看到当前开发任务的下一步动作，而不是在 Artifact、确认和交付入口之间来回找。

### CDW-10：多种交付方式

作为全栈开发工程师，我希望交付不仅支持本地目录，还能根据团队场景生成 GitHub PR、zip 包，或处理我手动上传的关键文件。

## 阶段目标

| 任务 | 目标 | 成功标准 |
|------|------|----------|
| TASK-020 | 服务端可信交付 | 写回前检测目标文件变化，失败有报告，关键动作可审计 |
| TASK-021 | 核心交互设计复盘 | 用户从 Project/Chat 能看懂下一步，并能顺畅进入 Artifact/Delivery |
| TASK-022 | 交付能力扩展 | GitHub PR、zip、upload 的授权边界和实现任务清楚 |

## 非目标

- 不做面向游客的营销首页。
- 不自动扫描用户本地目录或远程组织。
- 不跳过人工确认直接写入用户项目。
